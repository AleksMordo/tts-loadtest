# Журнал работ: нагрузочный стенд TTS для телефонии (2–4 июля 2026)

Хронология, проблемы, решения и текущее состояние проекта.
Метрики и таблицы — в [../results/report_a100_2026-07-03.md](../results/report_a100_2026-07-03.md);
здесь — полная картина «что происходило и почему».

---

## 1. Задача

Развернуть стриминговый TTS с zero-shot-клонированием голоса (изначально —
CosyVoice 2) для телефонии, определить сайзинг GPU в Yandex Cloud под
10/30/100 одновременных линий и посчитать стоимость секунды синтеза.

SLA: TTFB p95 ≤ 500 мс; RTF ≤ 0.8 на каждом потоке; 0 underruns на полке;
error rate < 0.1%; без роста VRAM; GPU util ≤ 85% в устойчивом режиме.

Definition of Done первого этапа: репозиторий, в котором вся логика проверена
end-to-end против локального mock-сервера, а Terraform/Ansible валидны.

## 2. Что построено (этап 1, без облака)

Репозиторий `tts-loadtest/`:

- **Terraform** (`terraform/yandex/`): VPC, security group (наружу только SSH
  с нашего IP), GPU-ВМ + CPU-ВМ генератора нагрузки в одной подсети.
  Размеры ужаты под дефолтные квоты облака (32 vCPU / 128 GB RAM / 200 GB SSD):
  GPU-ВМ 28 vCPU/119 GB/150 GB + loadgen 4 vCPU/8 GB/30 GB.
- **Ansible** (`ansible/`): роли common (sysctl, fd-лимиты, chrony), nvidia
  (драйвер + container toolkit), docker, tts (три бэкенда, см. ниже),
  monitoring (Prometheus + dcgm/node-exporter), loadgen. Inventory генерируется
  из `terraform output`.
- **Генератор нагрузки** (`loadgen/`, asyncio): модель «линии» — закрытый
  диалоговый цикл (реплика → стрим синтеза → пауза 1.5–4 c); метрики TTFB,
  RTF, underruns, inter-arrival пишутся JSONL-построчно; раннер исполняет
  матрицу сценариев (полки, ramp-up, поиск N_max шагом, spike) без участия
  человека. Транспорт клиента — абстракция с реализациями ws / grpc / triton.
- **Mock-сервер** (`mock_server/`) с параметризованной эмуляцией деградации
  GPU; e2e-тест проверяет, что раннер сам находит заложенный N_max = 20±2.
- **Отчёты** (`report/`): выгрузка GPU-метрик из Prometheus по окнам полок,
  cost-модель (цены только из pricing.yaml), сборка report.md + CSV.
- 23 pytest (юнит + e2e против mock), ruff, ansible-lint (production profile),
  terraform validate — всё зелёное. `make smoke-local` прогоняет весь пайплайн
  локально за пару минут.

Уточнения от заказчика по ходу: боевой протокол — gRPC; сервис собирается из
GitHub (Brakanier/FastCosyVoice), а не из private registry; выход 8 kHz —
скорректирован на фактические 24 kHz модели; референс-запись голоса
предоставлена (14.6 c + транскрипт).

## 3. Сессия 1 (3 июля, утро): python-gRPC сервер — тупик

Первый деплой на A100 (`gpu-standard-v3`). Штатный сервер форка
`runtime/python/grpc` + наши патчи.

**Проблемы и решения:**

| # | Проблема | Решение |
|---|---|---|
| 1 | `ubuntu-drivers --gpgpu` ставит только kernel-часть драйвера — нет `nvidia-smi` | роль nvidia доустанавливает `nvidia-utils-<ver>-server` |
| 2 | В requirements форка нет grpcio-tools — сборка образа падала | добавлен в Dockerfile |
| 3 | Ansible-шаблон: `trim_blocks` съел перевод строки, `ports:` приклеился к команде | однострочное jinja-выражение |
| 4 | zero-shot в сервере форка сломан: код передаёт тензор туда, где CLI ждёт файл, причём фронтенд читает его трижды | патч: tmp-WAV в /dev/shm, удаление после стрима |
| 5 | Сервер догенерирует отменённые запросы («зомби» жгут GPU после конца звонка) | патч: обрыв генерации по `context.is_active()` |
| 6 | Сток fp32: RTF 1.31 на одиночном потоке — медленнее реального времени | флаги `--fp16 --load_trt` (наш патч сервера) |
| 7 | dcgm-exporter крашлупится | `user: root` + SYS_ADMIN в compose |

**Итог сессии:** даже с fp16+TRT — TTFB 1.4 c, RTF 0.70 на одиночном потоке;
2 линии → RTF 1.42 и underruns; 4 линии → TTFB 11.6 c. При этом **GPU
загружен на 1.5–3.6%** (пик 15%): всё упирается в CPU/GIL питоновского
сервера (ONNX-токенизатор без CUDA, ThreadPool). **Вердикт: архитектура
непригодна для телефонии, N_max = 0 при любых деньгах.** Стенд погашен.

## 4. Сессия 2 (3 июля, день): Triton + TensorRT-LLM — стабильность

Переход на `runtime/triton_trtllm` (Triton Inference Server + TRT-LLM, decoupled
streaming). День глубокой отладки.

**Проблемы и решения:**

| # | Проблема | Решение |
|---|---|---|
| 8 | triton-каталог форка несовместим с образом `soar97/triton-cosyvoice:25.06` (HEAD форка переписан под cosyvoice3) | runtime из **апстрима** FunAudioLLM/CosyVoice, пин коммита эпохи образа (e8bf717) |
| 9 | В форке потерян `.gitmodules` — Matcha-TTS не клонируется, token2wav падает | явный clone Matcha-TTS в роли |
| 10 | **docker-proxy молча ломает decoupled-стрим**: изнутри контейнера работает, удалённым клиентам ответы не доходят | `network_mode: host` (подсказка была в их же README) |
| 11 | KV-кеш LLM 2560 токенов: русские тексты пробивают лимит, LLM виснет навсегда | 65536 токенов |
| 12 | **BLS ≥ 2 инстансов: запросы теряются на входе в tensorrt_llm, клин всего сервера** (upstream issue #1866, воспроизводится и на TRT-LLM 1.x) | сессия 2: BLS=1 (сериализация); позже собран свой образ на `tritonserver:25.12-trtllm` + перевод LLM в offline-режим — клин ушёл, но TTFB вырос |
| 13 | token2wav падает на хвостовом мел-чанке < 3 фреймов | патч: паддинг/пропуск хвоста |
| 14 | tritonclient aio «висел» — переписали на sync+callbacks, а sync дедлочился на stop_stream под нагрузкой | позже выяснилось: aio был невиновен — виноват docker-proxy (№10); транспорт возвращён на aio |

**Прорыв сессии:** BLS=1 на пине e8bf717: **TTFB 241–262 мс, RTF 0.10–0.23** —
оба SLA с запасом на одиночном потоке. Но конкурентность заблокирована багом №12.

**Стоп-фактор сессии:** у CosyVoice2-0.5B на русском рваная генерация,
детерминированно по тексту: «Здравствуйте, вы позвонили...» → **0.0 секунд
аудио**; другие фразы → обрывки 0.1–0.6 c или «болтовня» до токен-капа
(27–30 c на 9 слов). Русский референс не помог. Аудио-доказательства:
`results/triton_samples/`. **Мерить стоимость модели, которая не говорит
по-русски, бессмысленно** — стенд погашен, issue-hunt подтвердил (#1866 —
клин, #1790 — проблемы русского в v3 у одного репортёра).

## 5. Сессия 3 (3–4 июля, вечер/ночь): CosyVoice3 — успех

Решение заказчика: проверить CosyVoice3 (Fun-CosyVoice3-0.5B-2512, русский
заявлен официально с рейтингом «Good»).

Архитектура v3-деплоя апстрима (HEAD, пин 074ca6d) принципиально другая:
tritonserver (BLS cosyvoice3 + token2wav/vocoder, порты 18000-18002) +
**trtllm-serve** для LLM (порт 8000) — гоночная связка BLS↔tensorrt_llm-бэкенда
исключена архитектурно.

**Проблемы и решения:**

| # | Проблема | Решение |
|---|---|---|
| 15 | **xet-CDN HuggingFace (`us.aws.cdn.hf.co`) блокирован из YC** — загрузка виснет, остаются битые onnx/safetensors («DecodeError», «InvalidHeaderDeserialization») | `HF_HUB_DISABLE_XET=1` + перекачка |
| 16 | `x_transformers`/`s3tokenizer` обновляются скриптом только в prep-контейнере — serve падает (`No module named 'x_transformers'`, S3Tokenizer state_dict) | pip upgrade в обоих контейнерах (закреплено в compose) |
| 17 | **Матрицы: каждая линия делала ровно 1 реплику и умирала.** Корень: ruff-автофикс UP041 заменил `except asyncio.TimeoutError` → `except TimeoutError`, а на loadgen-ВМ Python 3.10, где это разные классы (слились в 3.11): таймаут паузы тихо убивал task линии. Локально (py3.12) не воспроизводилось. Это же объясняет «клин клиента» в матрицах сессии 2 | `except (TimeoutError, asyncio.TimeoutError)` + noqa; урок: target py311 в линтере при py3.10 на проде — мина |

**Результат — русский работает.** Прослушано и подтверждено заказчиком:
русская речь, клонирование голоса с 14-секундного референса адекватное.
Длительности фраз правдоподобны (4.3–7.2 c на 8–10 слов). Редкие «убеги»
генерации (единицы процентов, до 25 c) остаются.

**Полная матрица прогнана** (10/30/100 × cached/clone × 15 мин + N_max + spike):

- **GPU util 98–100% на всех полках** (против 2% у python-grpc) — стек
  архитектурно эффективен, узких мест в софте не осталось;
- VRAM стабильна (45.7 GB) — утечек нет;
- **пропускная способность константна: 6.53 секунды аудио в секунду** при
  любом числе линий — честный GPU-bound предел модели; лишние линии — очередь
  (TTFB p95: 4 c при 10 линиях → 12 c при 30 → 38 c при 100);
- одиночный TTFB ~900 мс — выше SLA 500 мс, поэтому строгий N_max = 0 —
  **следующий этап — тюнинг латентности**, ручки известны (стартовый
  token_hop, TRITON_MAX_BATCH_SIZE=1 в стоке для вокодера, приоритет
  префилла, BLS_INSTANCE_NUM);
- clone ≈ cached: в v3 BLS кеширует обработку референса (audio_tokenizer
  вызывался 10 раз на 570+ запросов).

## 6. Экономика (по фактическому биллингу аккаунта)

Юнит-цены из реального биллинга (июль 2026): A100 = 408.12 ₽/GPU·ч →
GPU-ВМ целиком + диск + IP = **C_hour = 475.62 ₽/час** (втрое дешевле
ориентира «~1017 ₽/час» из постановки задачи).

- **Секунда синтезированного аудио: 0.0202 ₽** (от duty_cycle не зависит)
- Ёмкость по throughput: 6.5 занятых линий = **16 линий при duty 0.4** на A100
- **Линия: 29.73 ₽/час; минута разговора: 0.495 ₽**
- 10 / 30 / 100 линий = 1 / 3 / 9 × A100 = 476 / 1 427 / 4 281 ₽/час
  (342 тыс / 1.03 млн / 3.08 млн ₽/мес)

Оговорка: ёмкость по throughput подразумевает очереди (мягкие SLA); после
тюнинга латентности SLA-ёмкость будет измерена отдельно и может быть ниже —
цена аудио-секунды при этом не изменится.

Стоимость всей проведённой работы в облаке: **6 728.62 ₽** (~14 GPU-часов,
три подъёма стенда).

## 7. Как всё работает сейчас

- `config/stand.yaml` — единый источник правды; **`tts.backend: triton_v3`**
  (дефолт) — CosyVoice3; `triton` (CosyVoice2, русский сломан) и `python_grpc`
  (тупик) оставлены для сравнения.
- `make up` — terraform + ansible: поднимает стенд с нуля до работающего
  CosyVoice3 (~1 час: драйвер, образ, веса с HF/ModelScope, TRT-LLM engine,
  два сервера, мониторинг, генератор). Все 17 граблей из таблиц выше учтены
  в ролях/патчах — повторный подъём их не встретит.
- `make test RUN_ID=...` — полная матрица в фоне на loadgen-ВМ;
  `make report RUN_ID=...` — GPU-метрики, отчёт, CSV; `make down` — гашение.
- Клиентский транспорт: tritonclient **aio**, модель `cosyvoice3`,
  порт 18001; свежий стрим на реплику.
- `config/pricing.yaml` заполнен реальными юнит-ценами из биллинга.
- Локальная проверка без GPU: `make lint` (23 теста) + `make smoke-local`
  (mock находит заложенный N_max) — по-прежнему зелёные.

## 8. Открытые хвосты (план следующей сессии)

1. **Тюнинг латентности** (цель — TTFB ≤ 500 мс): стартовый чанк BLS,
   батчинг token2wav/vocoder (в стоке batch=1!), kv fraction, приоритизация
   префилла в trtllm-serve. Затем полки 2–8 линий → честный SLA-N_max →
   финальная таблица стоимости при строгих SLA.
2. «Убеги» генерации (единицы %): ограничить max_tokens пропорционально
   длине текста в BLS.
3. Несколько референс-голосов (сейчас один — режимы cached/clone совпадают).
4. Спорадический повтор «двух tritonserver в одном контейнере» после
   `docker compose restart` — всегда использовать down/up (уже так в ролях).

## 9. Уроки (переносимые)

1. **«Заявлено в README» ≠ «работает»**: из четырёх серверных конфигураций
   форка/апстрима ни одна не завелась без патчей; язык из списка
   поддерживаемых оказался нерабочим в v2 и рабочим в v3.
2. **Всегда мерить утилизацию GPU**: 2% util при «загруженном» сервере мгновенно
   показали, что python-gRPC-путь мёртв, а 100% в v3 — что дальше тюнить модель,
   а не софт.
3. **docker-proxy и стриминг несовместимы тихо**: изнутри работает, снаружи
   виснет без единой ошибки. host-network для стриминговых серверов.
4. **Версионные пары считаются парами**: образ ↔ коммит скриптов (сессия 2),
   python на ВМ ↔ target линтера (проблема 17).
5. **Проверочная полка перед матрицей** каждый раз окупалась: ни одна матрица
   не была запущена на сломанной конфигурации дольше одной полки.
6. Эксперимент с заведомо известным ответом (mock c заложенным N_max, BLS=1
   бисекты, «short vs mixed») — самый быстрый способ локализовать слоёный баг.
