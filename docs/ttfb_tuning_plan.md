# План GPU-сессии: тюнинг TTFB → честный SLA-N_max

Цель: снизить одиночный TTFB с ~900 мс до ≤ 500 мс (SLA p95) на бэкенде
`triton_v3` (CosyVoice3) и найти **SLA-N_max** — сколько линий держит одна A100
при строгих SLA. Это уточнит экономику: 0.0202 ₽/сек из results/v3_full
посчитан по throughput (мягкие SLA), SLA-ёмкость будет ниже.

Бюджет: ~3–4 ч аренды (~1500–2000 ₽ по факту биллинга 475.62 ₽/ч).

---

## Из чего складывается TTFB (разбор model.py BLS, коммит 074ca6d)

Стриминговый путь `cosyvoice3` BLS: первый аудио-чанк уходит клиенту только
после того, как LLM нагенерит `token_hop_len + flow_pre_lookahead_len = 15 + 3
= 18` токенов (0.72 с аудио), затем token2wav (flow) + vocoder по первому чанку.

Слагаемые: prefill LLM (референс ~365 токенов + текст) → генерация 18 токенов
→ flow первого чанка → vocoder. Ручки, закодированные в stand.yaml:

| Ручка (stand.yaml) | Сток | Кандидаты | Что даёт |
|---|---|---|---|
| `v3_token_hop_len` | 15 | **8**, 5, 10 | Первый чанк после hop+3 токенов вместо 18 → главный рычаг TTFB |
| `v3_chunk_strategy` | exponential | time_based | Рост чанков после первого; влияет на RTF/стабильность, не на TTFB |
| `v3_bls_instance_num` | 10 | **10**, 4, 1 | Матрица v3_full шла с 1 (наследие клина v2!) — вероятная причина лесенки TTFB под нагрузкой. v3 клину не подвержен: LLM в отдельном trtllm-serve |
| `v3_kv_cache_fraction` | 0.4 | 0.4, 0.6 | Запас kv-кеша LLM при параллельных линиях |
| `v3_llm_max_batch_size` | 64 | 64 | Движок собран под 64 — выше нельзя, ниже незачем |

Все ручки применяются **без пересборки движка**: ansible патчит
`model_repo_cosyvoice3_copy` + run_cosyvoice3.sh и рестартует контейнер
(~2–3 мин на итерацию конфига).

## Риски / за чем следить

- **Малый hop → чаще вызовы flow/vocoder** — RTF может вырасти; underruns
  на полках 4–8 покажут. Если RTF ломается — откат hop до 10.
- **Короткий хвостовой чанк token2wav** (v2 падал, паддинг < kernel 3):
  на v3 с малым hop вероятность edge-кейса выше — смотреть error rate
  и лог tritonserver.
- **BLS=10 на v3 не проверен под нагрузкой** (пробовали только на v2, где был
  клин #1866). Если лесенка TTFB останется — пробовать 4/16.
- Vocoder в BLS пересчитывает весь накопленный mel на каждом чанке (O(n²) по
  длине реплики) — на длинных текстах RTF просядет независимо от hop; это
  кандидат на патч следующей сессии, в эту не лезем.

## Порядок прогона

### 0. Подготовка (до аренды, локально)
- [ ] Вписать в config/stand.yaml: свой `folder_id` (yc config get folder-id)
      и `allowed_ssh_cidr` (curl -4 ifconfig.me) — в репозитории они пустые.
- [ ] `make lint && make smoke-local` зелёные.
- [ ] `make up` → стенд (стадии 0–2 v3 ~30–60 мин — учесть в бюджете сессии).

### 1. Бейзлайн (сток, 1 конфиг, ~15 мин)
```
python -m loadgen.runner --set tune_scenarios --profile tune \
  --only tune_probe1 tune_probe2 tune_probe4 --run-id tune_baseline
```
Сток (hop 15, BLS 10): фиксируем warm TTFB p50/p95 на 1 линии и лесенку на 2/4.
Внимание: bls=10 — уже не как в v3_full (там был 1); если лесенка 2/4 исчезла
относительно v3_full — причина найдена, дальше только hop.

### 2. Итерации по hop (2–3 конфига × ~20 мин)
Менять `v3_token_hop_len` в stand.yaml → `make inventory deploy` (патч+рестарт)
→ пробы 1/2/4. Порядок: **8** → если TTFB всё ещё > 500: **5**; если RTF/underruns
сломались: **10**. Критерий: TTFB p95 (1 линия) ≤ 400 мс (запас на сеть до SBC),
0 underruns на 4 линиях.

### 3. (если лесенка осталась) BLS / chunk_strategy (~20 мин на вариант)
`v3_bls_instance_num`: 4 ↔ 10 ↔ 16; при болтанке RTF — `v3_chunk_strategy:
time_based`. Пробы 4/8.

### 4. Финал: SLA-N_max на лучшем конфиге (~1–1.5 ч)
```
python -m loadgen.runner --set tune_scenarios --profile cloud \
  --only nmax_fine_mixed_cached --run-id tune_final
python -m report.collect --run-id tune_final --prometheus-url http://<loadgen>:9090
python -m report.build_report --run-id tune_final
```
15-минутные полки, шаг 1 линия, старт с 1. N_max = последняя полка, где все SLA
выполнены (TTFB p95 ≤ 500, RTF max ≤ 0.8, 0 underruns, err < 0.1%).
