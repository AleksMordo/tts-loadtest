# Отчёт: нагрузочное тестирование CosyVoice2 на A100 (Yandex Cloud), 2026-07-03

## Конфигурация стенда

- GPU-ВМ: `gpu-standard-v3`, 1 × A100-SXM4-80GB (драйвер 595.71.05), 28 vCPU / 119 GB, SSD 150 GB
- Loadgen-ВМ: 4 vCPU / 8 GB, та же подсеть (внешняя сеть исключена из измерений)
- Модель: CosyVoice2-0.5B (ModelScope `iic/CosyVoice2-0.5B`), выход 24 kHz s16
- Затрачено аренды: ≈ 5–6 часов A100 (два подъёма стенда: python-grpc утром, Triton днём)
- Референс для zero-shot: синтетический (нагрузка честная, длительность генерации — см. ограничения) и реальный (речь из asset апстрима)

## Замеренные конфигурации сервинга

### 1. Штатный python-gRPC сервер (`runtime/python/grpc` + наши патчи)

Патчи: `--stream` (чанки по мере генерации), `--fp16 --load_trt`, tmp-wav для zero-shot
(оригинальный код сломан), обрыв генерации по отмене RPC.

| Режим | Линии | TTFB p95 | RTF max | Underruns | GPU util avg |
|---|---:|---:|---:|---:|---:|
| fp32 сток | 1 | 2 200 мс | 1.31 | — | — |
| fp16+TRT | 1 | 1 380 мс | 0.70 | 0 | ~0% |
| fp16+TRT | 2 | 3 004 мс | 1.42 | 19 | 1.5% |
| fp16+TRT, clone | 4 | 11 640 мс | 4.29 | 50 | 3.6% |

**Вердикт: N_max = 0 при SLA (TTFB p95 ≤ 500 мс).** Узкое место — CPU/GIL
(ONNX-токенизатор без CUDA, python ThreadPool): GPU простаивает (util ≤ 15% peak,
VRAM 47.7 GB), линии душат друг друга. Не лечится числом GPU.
Сырые данные: `results/diag/` (JSONL, meta, report.md).

## Найденные проблемы и статус

| # | Проблема | Статус |
|---|---|---|
| 1 | zero-shot в python-grpc сервере форка сломан (тензор вместо файла; файл читается фронтендом трижды) | починено (наш патч) |
| 2 | Сервер догенерирует отменённые запросы («зомби» после конца звонка) | починено (наш патч) |
| 3 | python-grpc: сток fp32 → RTF 1.3 на A100 | обойдено (fp16+TRT), но архитектурно тупик |
| 4 | triton-каталог форка Brakanier несовместим с образом soar97/triton-cosyvoice:25.06 | обойдено: runtime из апстрима, пин e8bf717 |
| 5 | docker-proxy молча ломает decoupled-стрим Triton для удалённых клиентов | починено: network_mode: host |
| 6 | KV-кеш LLM 2560 токенов — русские тексты пробивают лимит, LLM виснет | починено: 65536 |
| 7 | **BLS_INSTANCE_NUM ≥ 2: инстансы 2+ навсегда виснут на вызове tensorrt_llm** | **ОТКРЫТО — главный блокер** |
| 8 | LLM не ставит EOS на русском тексте: каждая фраза генерит 26–30 c аудио до токен-капа | открыто (вероятно, лечится русским референсом+транскриптом) |
| 9 | token2wav падает на коротком хвостовом чанке (padded input 2 < kernel 3) | открыто (редкий edge) |
| 10 | tritonclient aio не работает с decoupled-моделью (виснет) | обойдено: sync-клиент + callback |

## Выводы для сайзинга

1. Стек Triton + TensorRT-LLM на A100 достигает телефонных SLA на поток — вопрос
   ёмкости упирается в баг #7, а не в GPU.
2. До устранения #7 любые цифры N_max/стоимости канала преждевременны: при BLS=1
   формально N_max ≈ 1, «стоимость линии» ≈ полная стоимость ВМ.
3. Ожидание (не измерено!): с рабочей конкурентностью при RTF ~0.1 и inflight-батчинге
   LLM ёмкость одного A100 может составить десятки линий — проверяется матрицей
   сразу после фикса #7.

## Ограничения измерений

- Синтетический референс: модель «болтает» (26–30 c аудио на 10-словную фразу) —
  RTF на секунду аудио честен, но длительность занятости линии на реплику завышена.
  Для чистовых замеров нужны реальные русские записи 5–10 c + точный транскрипт (#8).
- Сеть до телефонии, ASR/LLM-часть диалога, трафик — вне теста.
- `pricing.yaml` не заполнен — стоимостные таблицы не считались.

Ключевые патчи (все в ansible-роли):
1. **LLM в offline-режиме** (`decoupled:False` только для tensorrt_llm) — гонка
   двух потоков BLS за IPC-канал стаба (decoupled-итератор LLM + унарные exec)
   теряла запросы даже на TRT-LLM 1.x. Аудио к клиенту стримится по-прежнему.
   Цена: TTFB включает полную генерацию токенов.
2. token2wav: паддинг/пропуск хвостового мел-чанка < 3 фреймов (краш вокодера).
3. Транспорт клиента: свежий gRPC-стрим на каждую реплику.

## Найденный блокер: качество CosyVoice2-0.5B на русском

Детерминированно по тексту (seed фиксирован): часть фраз даёт **пустой ответ**
(0 токенов), часть — **обрывок** (0.08–0.6 c), часть — **«болтовню» до
токен-капа** (27–30 c аудио на фразу из 9 слов). Звуковые доказательства:
`results/triton_samples/sample_normal.wav` (0.0 c на «Здравствуйте, вы
позвонили...») и `sample_short.wav` (14.8 c на 9-словную фразу).
Русский формально в списке поддерживаемых языков, но есть профильные issue
(например, #1790 — CosyVoice3 генерит китайский вместо русского).

**Вывод: измерять стоимость секунды синтеза бессмысленно, пока модель не даёт
валидное аудио на русском.** Это продуктовый вопрос выбора/доводки модели, не
инфраструктуры.

## Интерпретация

1. **GPU наконец полностью загружен (98–100%)** — v3-стек архитектурно эффективен,
   узких мест в софте больше нет. Это честный GPU-bound предел модели.
2. **Пропускная способность constant: ~580 реплик/15 мин (~0.64 rps) при любом числе
   линий** — лишние линии просто стоят в очереди (TTFB растёт линейно: 4 c → 12 c → 38 c).
   В пересчёте: система генерирует ~3+ секунды аудио в секунду → **эквивалент
   ~3 полностью занятых линий, или ~8 линий при duty_cycle 0.4**.
3. **Одиночный TTFB ~900 мс** — уже выше SLA 500 мс: N_max при строгих SLA = 0-единицы.
   Целевой SLA требует оптимизации латентности, а не только пропускной способности.
4. clone ≈ cached во всех полках: в v3 BLS работает кеш референса (audio_tokenizer
   вызывался 10 раз на 570+ запросов), а референс у нас один.

## Что тюнить дальше (в порядке ожидаемого эффекта)

1. **TTFB**: первый чанк = audio_tokenizer+embedding (кешируется) + LLM-префилл +
   первый хоп token2wav. Ручки: меньший стартовый token_hop, приоритизация префилла
   в trtllm-serve, тёплый кеш референса (у нас один голос — идеальный случай).
2. **Throughput**: батчинг token2wav/vocoder (TRITON_MAX_BATCH_SIZE=1 в стоке v3!),
   BLS_INSTANCE_NUM, kv_cache_free_gpu_memory_fraction 0.4 → выше.
3. «Убеги» генерации: ограничить max_tokens пропорционально длине текста.
4. Точный N_max: полки 2–8 линий после тюнинга TTFB.

# Экономика по фактическому биллингу (2026-07-04)

Юнит-цены восстановлены из реального биллинга аккаунта (20260604-20260731.csv):
A100 = 408.12 ₽/GPU·ч; итого GPU-ВМ (1×A100, 28 vCPU, 119 GB) + диск 150 GB + IP =
**C_hour = 475.62 ₽/час** (фактические списания). Весь тест (3 сессии, ~14 GPU-часов)
обошёлся в **6 728.62 ₽**.

Измеренный throughput CosyVoice3 на A100 (GPU util ~100%, константа на полках
10/30/100): **6.53 секунды аудио в секунду**.

## Стоимость (по throughput; мягкие SLA — очереди допустимы)

- **Секунда синтезированного аудио: 0.0202 ₽** (= C_hour / (3600 × 6.53) — честная
  цена аудио-секунды, от duty_cycle не зависит)
- Ёмкость: 6.5 полностью занятых линий = **16 линий при duty_cycle 0.4** на 1×A100
- **Линия/час: 29.73 ₽ | минута разговора: 0.495 ₽**

| Линий | Инстансов A100 (запас ×1.3) | ₽/час | ₽/мес (720 ч) | ₽/мин разговора |
|---:|---:|---:|---:|---:|
| 10 | 1 | 476 | 342 446 | 0.495 |
| 30 | 3 | 1 427 | 1 027 339 | 0.495 |
| 100 | 9 | 4 281 | 3 082 018 | 0.495 |

## Оговорки

1. Ёмкость посчитана **по пропускной способности**, а не по строгим SLA: при
   16 линиях/duty 0.4 очереди дадут TTFB в секунды. Для телефонного SLA
   (TTFB ≤ 0.5 c) нужен тюнинг латентности (см. сессию 3); после него SLA-ёмкость
   будет измерена полками 2–8 линий и, вероятно, окажется ниже 16 — стоимость
   линии соответственно выше. Цена аудио-секунды (0.0202 ₽) при этом не изменится,
   если тюнинг не уронит throughput.
2. Цены — фактические списания текущего аккаунта; прайс/скидки могут отличаться.
3. Redкие «убеги» генерации завышают audio_s_total на единицы процентов —
   цена секунды может быть на ~2-5% оптимистичнее.
