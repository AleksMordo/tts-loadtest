# Сессия тюнинга TTFB → SLA-N_max (2026-07-04/05, A100)

Цель: снизить TTFB со ~900 мс (v3_full, сток+BLS1) до SLA p95 ≤ 500 мс и найти
честную SLA-ёмкость одной A100. План: docs/ttfb_tuning_plan.md.

SLA: TTFB p95 ≤ 500 мс; RTF ≤ 0.8 на каждом потоке; 0 underruns; err < 0.1%.
Нагрузка: mixed-тексты (80% short / 20% long), voice=cached (один референс,
кеш спикера в BLS), профиль tune = 3-мин полки, финал = 15-мин полка (cloud).

## Итог

**SLA-N_max = 2 линии на A100** (подтверждено 15-мин полкой, см. tune_final).
**Тёплый одиночный TTFB p50/p95 = 265/270 мс** (было ~900) — SLA с двукратным
запасом. 3 линии — почти (p95 558, +12% к SLA), 4+ — очередь.

Финальный конфиг (stand.yaml):
```yaml
v3_token_hop_len: 15        # сток; hop<12 рвёт стык чанков 1->2 (underruns)
v3_chunk_strategy: exponential   # time_based — регресс (см. итерацию B)
v3_bls_instance_num: 10     # сток; v3_full шёл с 1 — источник старой лесенки
v3_token2wav_instances: 8   # сток 1 — сериализовал flow между линиями
v3_vocoder_instances: 8     # сток 1
v3_kv_cache_fraction: 0.4   # сток
v3_llm_max_batch_size: 64   # сток
```
**+ короткий референс ~6 с вместо 14.6 с — главный рычаг** (см. ниже).

## Разбор TTFB (BLS model.py, 074ca6d)

Первый аудио-чанк уходит клиенту после `token_hop_len + 3` LLM-токенов, затем
flow (token2wav) + vocoder первого чанка. Ключевая неочевидность: **референс
прогоняется через flow на каждом чанке** (prompt mel-кадры) и сидит в prefill
LLM — длина референса напрямую сидит и в TTFB, и в стоимости каждого чанка.

## Хронология итераций (пробы = тёплые 3-мин полки)

| Итерация | Конфиг | 1 линия p50/p95 | 2 линии p50/p95 (undr) | 4 линии p50/p95 (undr) |
|---|---|---|---|---|
| baseline2 | сток hop15, BLS10, реф 14.6s | 517/580 | 512/849 (8) | 878/1419 (64) |
| A | hop 8 | 375/746 (36 undr!) | 384/834 (60) | 857/1624 (117) |
| B | + time_based, t2w/voc ×4 | 738/846 (96!) | 621/828 (408!) | 870/1127 (344!) |
| C | hop 10, exponential, t2w/voc ×4, **реф 6s** | 265/665¹ | 256/427 (9) | 372/637 (56) |
| D | hop 12 | **265/270 (0)** ✅ | 266/470 (3) | 378/668 (16) |
| E | hop 15, t2w/voc ×8 | — (тёплый ✅) | **294/465 (0)** ✅ | 405/779 (5) |
| E | 3 линии | | 3 линии: p95 558 (0 undr) — fail только по TTFB | |

¹ p95 итерации C на 1 линии — прогрев: первые 60 с после рестарта p50 627,
дальше p50/p95/max = 263/269/274. Кеш спикера греется в каждом BLS-инстансе
отдельно — в проде решается прогревочным запросом при старте/деплое.

Выводы по ручкам:
- **Референс 14.6→6 с: p50 −250 мс** (517→265). Продовая рекомендация: референсы
  5–8 с, обрезка по границе фразы, транскрипт обязан совпадать с аудио.
- **hop < 12 при exponential рвёт стык чанков 1→2**: фора первого чанка
  (hop/25 с) должна покрывать LLM(28 ток)+flow+vocoder следующего (~400-500 мс
  под конкуренцией). hop 15 (сток) = 600 мс форы = 0 underruns до 2 линий.
- **time_based — регресс**: мелкие чанки → много вызовов flow, а vocoder в BLS
  пересчитывает весь накопленный mel на каждом чанке (O(n²) по длине реплики) —
  RTF>1 даже на 1 линии. Кандидат на патч: инкрементальный vocoder.
- **t2w/voc инстансы ×4..8 + BLS 10**: убирают сериализацию до 2-3 линий;
  дальше упирается LLM-конкуренция (p50 растёт на 4+ линиях) — лечится только
  вторым trtllm-serve/GPU либо тюнингом батчинга LLM.
- Качество: с реальным русским референсом+транскриптом обрывков и болтовни НЕТ
  (0 err, 0 коротких на 300+ реплик). 30-сек реплики = длинные тексты, упёршиеся
  в max_tokens 750 (≈30 с аудио) — для телефонии не критично, кап поднимается
  в model.py BLS при необходимости.
- Синтетические референсы (gen_ref_audio) дают болтовню/обрывки — только для
  smoke на mock, на GPU-стенде использовать реальные записи.

## Финальное подтверждение (15-мин полка, 2 линии, cloud-профиль)

- 408 реплик, 0 ошибок, **0 underruns**
- TTFB p50/p95/p99 = **294/476/575 мс** — SLA p95 ≤ 500 выполнен
- RTF max 0.36 (SLA ≤ 0.8 с запасом)
- Синтезировано 4178 с аудио за 960 с полки (2 линии)
- GPU util avg ≈ 35%, VRAM max 68.5 GB (из них ~22 GB — 8 инстансов token2wav)

**GPU при SLA-нагрузке недогружен втрое** — ёмкость ограничена латентностью
конвейера (LLM-конкуренция на 3+ линиях), а не мощностью A100. Главный резерв
дальнейшего роста N_max — конкурентность LLM (профилирование trtllm-serve
батчинга, отдельный LLM-инстанс, или меньшая карта под линию).

SLA-N_max — путь известен (LLM-конкуренция), это задача следующего этапа.

## Инфраструктурные заметки сессии

- nvcr.io отдаёт 403 из сетей YC на любой тег — dcgm-exporter переведён на
  зеркало Docker Hub (docker.io/nvidia/dcgm-exporter:4.5.2-4.8.1-ubuntu22.04);
  метрики DCGM_FI_DEV_GPU_UTIL/FB_USED в 4.x на месте.
- GPU-ВМ (A100) грузится дольше loadgen — первый ansible-заход может не достучаться
  по ssh; второй запуск плейбука идемпотентно доводит.
- Все ручки тюнинга применяются ansible-патчами model_repo_copy без пересборки
  движка, итерация конфига = ~3 мин (deploy+restart) + прогрев.
